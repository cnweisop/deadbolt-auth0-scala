Deadbolt and Auth0

This starter application demonstrates how to use Deadbolt to provide authorization and drive authentication triggers. The identity management platform Auth0 is used to provide authentication support.

This example was written as part of the Deadbolt user guide.

AuthSupport

The bulk of the security mechanism can be found in AuthSupport. This manages the retrieval of the subject from the request, the cache or Auth0 in that order, as well as various other functionality. This class can essentially remain unchanged in most applications, unless some form of local user storage such as a database is also used.

AuthController

The entrypoint for a callback from Auth0. Once a user has authenticated on the identity management platform, the application is notified and provided with an authentication code for further interactions.

MyDeadboltHandler

This drives the authentication behaviour based on the requirements of the application's authorizations. If (for example) a subject is not required to access a resource and no subject is present, access will be granted; if (also for example) a subject is required to access a resource but is not present, an unauthorised response (with the login page as content) will be sent back. In cases where a subject must hold certain roles or permissions to access a resource and they do not meet the requirements, the client will receive a forbidden response.

login.scala.html

A very simple page containing the script provided by Auth0 to generate a log-in dialog.
